Core Vision of the ProjectThe core vision is to build a 100% private, locally-hosted, multi-format repository intelligence engine that unified code, database schemas, deployment infrastructure, and business documentation into a singular, semantic knowledge graph.Unlike standard document QA systems, this platform treats an entire workspace folder not as a disconnected pile of text chunks, but as a living system of highly interdependent components. By mapping structural code paths alongside unstructured documentation text, it provides deep impact-analysis, architectural reasoning, and contextual search completely offline.3. Detailed Explanation of Each FeatureMulti-Format Structural Ingestion EngineStandard tools ingest plain text uniformly. This feature runs specialized, layout-aware parsers that process code files, structured tables, and media formatting differently. Markdown-aware extraction preserves python indentation structures, JSON structural scopes, and SQL creation semantics. It isolates tables inside documents as individual blocks rather than mixing them into running paragraphs.Anthropic-Style Contextual Retrieval ProcessingWhen standard RAG chops files into distinct paragraphs, individual fragments lose reference scope. If a chunk reads, "Set maximum limit to 50," it becomes meaningless in isolation. This engine uses a fast local model pass during ingestion to prepend a 1-sentence global contextual summary to every child chunk (e.g., [Context: From auth.py config handling login security metrics]).Hierarchical Parent-Child Data ChunkingInstead of forcing a single chunk size compromise, the engine indexes small child chunks (e.g., 150-200 tokens) for highly precise mathematical matching. When a match is triggered, it passes the larger parent chunk block (e.g., 800-1000 tokens) containing full context, classes, or paragraph loops to the generating LLM.Deterministic Metadata Citation PreservationCitations are hardcoded directly into database rows at the exact moment of chunk creation. This locks absolute tracking variables (file_path, line_start, page_number, created_at timestamp) to the data layout. The synthesis engine is forced to draw citations exclusively from this metadata rather than guessing sources.Hybrid Search ArchitectureTo catch both abstract concepts and exact identifiers, the system queries data via two parallel pathways:Dense Embeddings: Evaluates overall conceptual meaning (e.g., matching "session timeout" with "login loops").Sparse BM25 Tokens: Performs exact keyword matching for precise structural tokens (e.g., USER_ID_7A).RRF Fusion & Neural Cross-Encoder RerankingThe dual outputs of the dense and sparse search streams are unified using Reciprocal Rank Fusion (RRF), which scores items based on their relative rank position across both lists without needing complex score normalization. The top results are then passed through a local Cross-Encoder Reranker network that calculates a deep, interactive match score between the user query and the full text snippet.Lazy/On-Demand GraphRAG Extraction EngineInstead of paying massive compute costs to map an entire workspace into a database upfront, the system uses a lazy extraction schema. It builds out a traditional fast hybrid index first. When a complex structural query is detected, it isolates the specific entities involved and extracts localized relationship graphs on-the-fly, reducing graph creation costs by up to 80%.3-Tier Adaptive Query Router Guard LayerA lightweight Pydantic classification loop monitors every incoming request before a single search path is fired. It separates inputs into three distinct pipelines:Direct Response: For baseline greetings or contextless inputs.Single-Hop RAG: For specific single-document lookups.Multi-Hop Complex Loop: For intensive, multi-document structural tracing.Interactive Multi-Turn Session Memory SystemTracks past dialogue trees and parameter details across conversational turns. It rewrites subsequent vague user requests (e.g., "What about its configuration?") into fully explicit retrieval prompts based on active historical references.AI-Assisted Impact Analysis Advisory ConsoleA specialized read-only reasoning view designed to track dependency footprints. When a structural change query is processed, it synthesizes an end-to-end trace mapping how altering a database variable shifts backend dependencies and alters documentation blocks.Local-First Offline Execution LayerThe entire application infrastructure operates on a localized hardware boundary via a private loopback address (localhost:11434). This keeps all processing entirely offline, ensuring no proprietary data leaves the local workstation.7. End-to-End User Workflow[Phase 1: Ingestion Pipeline]

Retrieval & Search Concepts DiscussedCommodity RAG applications look at documents through singular similarity lenses. This system deploys a structural Three-Stage Search Matrix:$$\text{Raw Content} \longrightarrow \text{Hybrid Search (Dense + BM25)} \longrightarrow \text{RRF Fusion Matching} \longrightarrow \text{Cross-Encoder Filtering}$$The dense matching component catches abstract conceptual phrases. The sparse token mechanism ensures exact keyword matching for specific variables or column definitions. By combining both approaches through Reciprocal Rank Fusion (RRF), the engine addresses the classic failure mode where vector search surfaces text that looks vaguely similar but lacks the exact identifier the user needs.10. Contextual Retrieval IdeasTo prevent chunks from losing context, this engine uses a specialized pre-indexing workflow inspired by Anthropic's Contextual Retrieval architecture.During ingestion, the full parent document is parsed, and a light local LLM summarizes its global context in one sentence. This summary statement is then dynamically prepended to every single child text block generated from that source.+-----------------------------------------------------------------------------------------+
| [PREPENDED CONTEXT SUMMARY]:                                                            |
| "This block belongs to user verification routines inside auth.py using SQL definitions" |
+-----------------------------------------------------------------------------------------+
| [RAW CONTENT SNIPPET]:                                                                  |
| "if user_status_id == 3: return AccessDeniedError"                                      |
+-----------------------------------------------------------------------------------------+
This prevents the model from isolating a block of code and losing track of what file, module, or business function it serves.11. GraphRAG / Relationship Mapping ConceptsRather than indexing every possible entity relationship up front—which is computationally expensive and introduces unnecessary complexity—this architecture uses a Lazy/On-Demand Graph Extraction Engine.The graph database (Neo4j) functions as an indexing tool focused on mapping high-value architectural nodes:[SQL Database Column] ──(Queried By)──► [Python Backend File] ──(Documented In)──► [Slide Component]
By prioritizing technical connections (like function imports, SQL usage, and directory trees) over generic text links, the graph engine avoids unnecessary bloat and maps dependencies with high structural accuracy.12. Impact Analysis System ConceptThe system is built as a Read-Only Impact Analysis Advisor, focusing on tracking structural dependencies rather than automatically rewriting your files.When a user asks what will break if a component changes, the engine maps the dependency footprint across your files and documentation. It then presents a clear impact report in the terminal UI, showing you exactly where the updates need to be made while prompting the user to verify the changes manually.13.
